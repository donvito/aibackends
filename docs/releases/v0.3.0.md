# aibackends v0.3.0 — Runtime Reuse, Preloading, and Benchmarks

This release eliminates the biggest source of latency in repeated task calls:
model reloading. Runtimes are now cached process-wide, models can be preloaded
at startup, and a new benchmark suite documents the gains with committed,
reproducible reports.

## Highlights

### Warm task calls, out of the box

Previously every `task.run()` built a fresh runtime and reloaded the model
from scratch. Runtimes are now cached process-wide and reused across task
calls, so the model loads once and every subsequent call only pays for
inference. No code changes required — it is on by default.

Measured impact (10+ samples per scenario, reports in `benchmarks/reports/`):

| Setup | Fresh runtime per call | Warm call | Speedup |
|---|---|---|---|
| llama.cpp, gemma4-e2b, RTX 3090 | 2,758 ms | 73 ms | 37.7x |
| transformers, gemma3-270m-it, CPU | 10,079 ms | 909 ms | 11.1x |

Larger models gain the most: llama3-8b loads for ~72 s and qwen3-vl-8b for
~91 s — costs that are now paid once per process instead of on every call.

### `preload()` — pay the load cost at startup

```python
import aibackends

aibackends.preload(runtime="llamacpp", model="gemma4-e2b")
# first task call is already warm
```

`BaseRuntime.preload()` is available for custom runtimes (no-op by default),
mirroring the existing PII `backend.load()` pattern.

### Opt-out flag for a zero-risk rollout

Set `reuse_runtime=False` globally (`aibackends.configure(...)`) or per call
to restore the previous build-per-call behavior exactly.

### `clear_runtime_cache()`

Cached models stay resident in RAM/VRAM by design. Call
`aibackends.clear_runtime_cache()` to release them; `reset_config()` clears
the cache as well.

### openai-privacy PII backend caching

The `openai-privacy` backend previously rebuilt its token-classification
pipeline on every call. It now caches the pipeline process-wide (matching
GLiNER) and supports `backend.load()` for explicit pre-warming. For
reference, warm GLiNER redactions run ~29x faster than the first
load-inclusive call (454 ms vs 13.4 s on the benchmark machine).

### Benchmark suite with committed reports

New `benchmarks/` directory with dated markdown reports in
`benchmarks/reports/`:

- `benchmark_runtime_reuse.py` — fresh-per-call vs cached runtime vs
  `preload()`.
- `benchmark_tasks.py` — per-model task latency (summarize, classify,
  extract, embed) plus VL describe-image benchmarks (llama.cpp, Gemma and
  Qwen VL models). Pass `all` to benchmark every recommended catalog model.
- `benchmark_pii_backends.py` — PII model load vs warm redactions.
- `run_all.py` — runs every benchmark sequentially, one process at a time; a
  shared lock file prevents concurrent runs from contending for a single GPU.

Reports include consistency statistics (stdev, CV, p50/p95, drift) and
per-segment trends. Across all models benchmarked, drift over sustained runs
was flat to negative — no inference degradation from runtime reuse.

## Behavior changes

- `get_runtime(...)` returns a shared cached instance for the same
  load-affecting config (runtime, model, device, ...). Per-call generation
  parameters (temperature, max_tokens, callbacks) are still applied fresh on
  every call.
- `LlamaCppRuntime` and `TransformersRuntime` serialize model loading and
  inference behind a per-instance lock, since shared instances may now be
  reached from multiple threads. Concurrent calls to the same model queue
  rather than reload.
- Loaded models remain in memory until `clear_runtime_cache()` /
  `reset_config()` or process exit.

## New public API

- `aibackends.preload(...)`
- `aibackends.clear_runtime_cache()`
- `RuntimeConfig.reuse_runtime: bool = True`
- `BaseRuntime.preload()`
- `aibackends.backends.pii.openai_privacy.load_privacy_pipeline` /
  `clear_pipeline_cache`

## Known issues found while benchmarking

- Three catalog model IDs no longer exist on Hugging Face and fail to load:
  `gemma3-4b`, `gemma3-12b`, `phi4-mini` (all pointing to removed
  `bartowski/*-GGUF` repos). To be repointed in a follow-up.
- `classify`/`extract` structured output fails schema validation on
  gemma4-e2b, gemma4-e4b, and mistral-7b under llama.cpp JSON mode
  (model-quality issue, unrelated to caching).
- The `bge-small` and `minilm-l6` embedding profiles are transformers-only;
  they point to non-GGUF repos and cannot run under llama.cpp.
